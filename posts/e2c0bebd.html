2、概要

Spring是非常流行和成功的Java应用开发框架，Spring Security正式Spring家族中的成员。Spring Security是基于Spring的框架，提供了一套安全性完整的解决方案。

正如你可能知道的关于安全方面的两个主要区域是认证和授权（或者访问控制），一般来说，Web应用的安全性包括 用户认证和用户授权两个部分，这两点也是SpringSecurity重要核心

  • 用户认证：验证某个用户是否为系统中的合法主体，也就是说用户能否访问该系统，用户认证一般要求用户提供用户名和密码， 系统通过校验用户名和密码来完成认证过程。通俗点说，就是系统认为用户是否能登录
  • 用户授权指的是验证某个用户是否有权限执行某个操作。在一个系统中，不同用户所具有的权限是不同的。比如对一个文件来说，有的用户只能进行读取，而有的用户可以进行修改。一般来说，系统会不同用户分配不同的角色，而每个用户角色则对应一系列的权限。通俗点讲就是系统判断用户是否有权限去做某些事情。

应用场景：在电商项目中，用户只能对商品进行购买和查看，而管理员可以队商品进行增删改查等操作。这就是用户权限，给不同用户给与不同的权限

3、Spring Security的历史

Spring Security开始于2003年年底，Spring的acegi安全系统。起因是Spring开发者邮件吗列表中的一个问题，有人提问能否考虑提供一个基于Spring的安全实现。

spring security以The Acegi Security System for Spring的名字其实于2013年晚些时候，一个问题提交到Spring开发者的邮件列表，询问是否已经有考虑一个基于Spring的安全社区实现 ，那时候Spring的社区相对较小（相对于现在）。实际上Spring自己在2013年只是一个存在与ScourseForge的项目，这个问题的回答是一个值得研究的领域，虽然目前时间缺乏组织了对他的探究

考虑到这一点，一个简单的安全实现建成但是并没有发布。几周后，Spring社区的其他成员询问了安全性，这次这个代码被发送给他们。其他几个请求也随之而来。到2004年1月大约有20万人使用了这个代码。这些创业的人提出了一个SourceForge项目加入是为了，这是在2004年三月成立的

在早些时候，这个项目还没有任何自己的验证，模块，身份验证过程依赖于容器管理的安全性和Acegi安全性。而不是专注于授权，开始的时候这很合适，但是越来越多的用户请求额外的容器支持 ，容器特定的认证领域接口的基本实现限制变得清晰。还有一个相关的问题增加新的容器路径，这里是最终用户的困惑和错误配置的常见问题

Acegi安全特定的认证服务介绍，大约一年以后，Acegi安全正式成为了Spring框架的子项目，1.0.0最终版本是出版于 2006，在超过两年半大的大量的软件生产的软件项目和数以计百的改进和积极利用社区的贡献

Acegi安全2007年正式成为了Spring组合项目，更名为Spring Security

4、同款产品对比

4.1、SpringSecurity

Spring技术栈的组成部分

通过提供完整可扩展性的认证和授权支持保护你的应用程序

spring security的特点：

  • 和Spring无缝结合
  • 全面的权限控制
  • 专门为Web开发而设计的
    • 旧版本不能脱离Web环境使用
    • 新版本队整个框架进行了分层抽取，分成了核心模块和Web模块。单独引入核心模块就可以脱离Web环境了
  • 是一个重量级框架，需要引入其他的东西

Shiro

是阿帕奇下的轻量级权限控制框架

特点：

  • 轻量级。shiro主张的理念是吧复杂的事情变得简单。针对性能有更高的要求，的互联网应用有更好的表现
  • 通用性
    • 好处：不局限于Web环境，看了一可以脱离Web环境使用
    • 缺陷：在Web环境下一些特定的需求需要手动编写代码定制

4.3、两者比较和具体使用

Spring Security是Spring家族中的一个安全管理框架，实际上，在Spring Boot出现之前，SpringSecurity就已经发展了许多年，但是使用的并不多，安全管理这个领域，一直是Shiro的天下

相对于Shiro，在SSM中整合Spring Security都是比较麻烦的操作，所以Spring Security随 功能比shiro强大，但是使用反而没有Shiro多（Shiro虽然功能没有Spring Security多，但是对于大部分而言，Shiro也够用了）

自从出了SpringBoot之后，Spring Boot对于Spring Security提供了自动化方案，可以使用更少的配置来管理Spring Security，

==因此，一般来说，常见的安全管理技术栈的组合为如下方式==

  • SSM + Shiro
  • Spring Boot / Spring Cloud + Spring Securyity

以上只是一个推荐的组合，如果单纯从技术上来说，无论怎么组合，都是可以运行的

2、Spring Security的基本原理

Spring Security本质上是一个过滤器链，总共有十几个过滤器形成了过滤器链

从启动是可以获取到过滤器链的

下面介绍过滤器链中的其中三种过滤器链：

  • FilterSecurityInterceptor：是一个方法级的权限过滤器，基本位于过滤器链的最底部
  • ExceptionTranslationFilter：是个异常过滤器，用来处理在认证授权过程中抛出的异常
  • UsernamePasswordAuthenticationFilter：对/login的POST请求做拦截，校验表单中用户名和密码

4、基于角色或权限进行访问控制

hasAuthority 和 hasAnyAuthority 、hasRole 和 hasAnyRole

注意：hasAuthority 、hasAnyAuthority 中的条件是且的关系，hasAnyAuthority ，和 hasAnyRole 需要满足的条件是或的关系

4.1、hasAuthority 方法

如果当前的主体具有指定的权限，则返回 true,否则返回 false，只能指定一个用户的权限，如果想要指定多个用户的权限，则需要使用hasAnyAuthority 方法,【用户可以设置多个权限，只要有一个权限满足了设置的权限就可以访问】

4.2、hasAnyAuthority 方法

hasAnyAuthority 方法相比与hasAuthority方法相比，他可以设置多个用户的权限，【其中满足权限的条件是或的关系，只要用户满足其中一个权限就满足条件了】，当一个用户既具有普通用户的权限，又具有管理员用户的权限的时候，这时候，使用hasAuthority 方法就不能解决方法了

hasAnyAuthority 传入多个参数

【方法一：直接将多权限设置为一个字符串，多个权限之间使用 , 隔开。例如：hasAnyAuthority ( “admin,user” )】

【方法二：因为hasAnyAuthority 中的参数为可变形参，所以每个权限都可以是一个参数，例如：hasAnyAuthority( “admi” , “user”) 】
